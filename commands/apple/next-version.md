# Start Next Version

Begin planning the next version of your app after completing a milestone.

## When to Use

- After `/apple:milestone` completes a version
- Starting work on v1.1, v2.0, etc.
- Planning a major update

## Step 1: Load Context

Read existing files:

```
Read: .planning/APP.md              # App specification
Read: .planning/STATE.md            # Current state
Read: .planning/IDEAS.md            # Captured ideas (if exists)
Read: .planning/archive/            # Previous milestones
```

Check for previous milestone:

```bash
ls -la .planning/archive/
```

## Step 2: Determine Version

If version provided as argument, use it. Otherwise ask:

```
What version are you planning?

Previous: v[LAST_VERSION]

Options:
- Patch (1.0.1) - Bug fixes only
- Minor (1.1.0) - New features, backwards compatible
- Major (2.0.0) - Breaking changes or major overhaul
- Custom - Enter your own version
```

## Step 3: Gather Version Scope

### Review Deferred Items

If previous MILESTONE.md exists, show deferred items:

```
Deferred from v[LAST_VERSION]:
- [ ] [Item 1]
- [ ] [Item 2]

Include these in v[NEW_VERSION]?
```

### Review Captured Ideas

If IDEAS.md exists:

```
Captured ideas:
- [ ] [Idea 1] - [date captured]
- [ ] [Idea 2] - [date captured]

Include any in v[NEW_VERSION]?
```

### Gather New Requirements

Ask about new version goals:

```
What's the focus for v[NEW_VERSION]?

1. Bug fixes from user feedback
2. New features (describe)
3. Performance improvements
4. Platform updates (new iOS/macOS version)
5. Design refresh
6. Other
```

### Review TestFlight/App Store Feedback

```
Any feedback from users to address?

- App Store reviews
- TestFlight feedback
- Support emails
- Crash reports
```

Generate new roadmap based on version scope:

```markdown
**Started:** [date]
**Target:** [target date if known]
**Previous:** v[LAST_VERSION]

## Version Goals

[Summary of what this version aims to achieve]

### Phase 1: [Name]
**Status:** Not Started
**Objective:** [What this phase accomplishes]

Tasks:
- [ ] [Task 1]
- [ ] [Task 2]

### Phase 2: [Name]
**Status:** Not Started
**Objective:** [What this phase accomplishes]

Tasks:
- [ ] [Task 1]
- [ ] [Task 2]

[Continue as needed...]

---

## Carried Forward

Items from previous version:
- [ ] [Deferred item 1]
- [ ] [Deferred item 2]

---

## Out of Scope

Explicitly NOT in this version:
- [Item 1] - Reason
- [Item 2] - Reason

---

## Success Criteria

v[NEW_VERSION] is complete when:
- [ ] [Criterion 1]
- [ ] [Criterion 2]
- [ ] All phases complete
- [ ] /apple:verify passes
- [ ] /apple:review passes

---

*Generated by SwiftShip /apple:next-version*
```

## Step 6: Update APP.md (if needed)

If version scope changes app definition:

```
Does v[NEW_VERSION] change the app's core definition?

- New platforms (adding watchOS, macOS)?
- New major features that change the concept?
- Different monetization?
- New Apple frameworks?

If yes, update APP.md. If no, keep existing.
```

## Step 7: Clear Working Files

Reset files that are version-specific:

```bash
# Remove old PLAN.md (will be regenerated)
rm .planning/PLAN.md 2>/dev/null

# Remove old VERIFICATION.md
rm .planning/VERIFICATION.md 2>/dev/null

# Keep REVIEW.md if issues should carry forward
# Keep ASO.md (usually doesn't change much)
# Keep DEPLOY.md (deployment config persists)
```

## Step 8: Completion

```
v[NEW_VERSION] planning started!

Created/Updated:
- .planning/ROADMAP.md - New version roadmap
- .planning/STATE.md - Reset for new version

Version scope:
- Phases: [N]
- Carried forward: [N] items
- New features: [N]

Next steps:
1. Review .planning/ROADMAP.md
2. Run /apple:discuss 1 to set preferences
3. Run /apple:plan 1 to detail Phase 1
4. Run /apple:build to start implementing

Let's build v[NEW_VERSION]! 🚀
```

## Version Planning Tips

### Patch Release (x.x.1)
- Focus: Bug fixes only
- Phases: Usually 1-2 (Fix, Test)
- Timeline: 1-2 weeks

### Minor Release (x.1.0)
- Focus: New features, improvements
- Phases: 3-5 typical
- Timeline: 4-8 weeks

### Major Release (x+1.0.0)
- Focus: Significant changes
- Phases: 5-7 typical
- Timeline: 2-4 months
- Consider: Breaking changes, migration
